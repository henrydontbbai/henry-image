# Henry Image Failure Policy

Start with `workflow-map.md` for the overall flow. This page only owns blocker classification and recovery.

Failures are split into API failures and quality failures.

## API Failures

Classify these precisely:

- `missing_credentials`: no usable local API key.
- `invalid_credentials`: key rejected by provider.
- `unsupported_router`: route accepts chat but rejects image generation.
- `unsupported_route`: one image route fails, but another route may still work (`responses` vs `images`).
- `bad_parameter`: invalid model, size, quality, background, mask, or file id.
- `rate_limited`: provider throttled the request.
- `quota_exceeded`: provider reports quota or billing exhaustion.
- `content_policy`: request was blocked by policy.
- `network_error`: DNS, connection, TLS, proxy, or transport failure.
- `timeout`: request timed out.
- `stream_disconnected`: Codex/Responses stream disconnected before the image turn completed.
- `outer_timeout`: the shell/tool runner killed the CLI before the CLI request timeout elapsed.
- `child_no_result`: a detached job process exited before writing a valid CLI envelope.
- `child_invalid_json`: a detached job wrote stdout but not a valid CLI JSON envelope.
- `cancelled`: Henry job was intentionally cancelled.
- `cancel_failed`: conservative cancellation could not terminate one or more recorded PIDs.
- `not_running`: no active recorded PID was found when cancellation was requested.
- `server_error`: upstream 5xx or retryable provider failure.
- `no_image_result`: request completed but no image bytes were returned.
- `write_error`: generated bytes could not be saved.

For API failures:

- state the exact blocker;
- do not ask Henry to paste secrets in chat;
- do not silently switch paid providers, proxy routes, or model families;
- for text-to-image generation, `--route auto` may try Responses first and then `/images/generations` on route-level failures such as `no_image_result`, unsupported tools, 404/405, timeout, or upstream 5xx;
- do not route/provider fallback on invalid credentials, missing credentials, content-policy, quota, rate-limit, or bad-parameter failures; explicit provider/model-family changes require Henry's choice;
- return a prompt package only when real image bytes cannot be produced after the checked CLI route fails, or when Henry explicitly asked for prompt-only output.

## Failure quick actions

| Status/code | Action |
| --- | --- |
| `stream_disconnected` | Switch complex work to CLI `--background-job`. |
| `outer_timeout` | Use background jobs or increase the outer shell/tool timeout above CLI `--timeout`. |
| `missing_credentials` / `invalid_credentials` | Stop and inspect redacted `auth_plan`, `auth_shape`, `header_names`, `query_names`, `provider_family`, and `adaptive_reason`; do not ask Henry to paste secrets or cross provider automatically. |
| `rate_limited` / `quota_exceeded` | Stop or use only an explicitly authorized provider; do not broad-fallback. |
| `content_policy` | Revise prompt; do not provider-fallback. |
| `bad_parameter` | Adjust model/size/quality/background; do not route/provider-fallback. If the error is only an unsupported Image API field, explicitly try `--images-compat minimal`. |
| `unsupported_router` / `no_image_result` | Use CLI `--route auto` / Image API fallback first; prompt package only after checked CLI failure or explicit prompt-only request. |
| `child_no_result` / `child_invalid_json` | Report `stderr_tail`, `candidate_attempts`, and job metadata; retry only when the same route is still appropriate. |
| `cancelled` | Start a new background job only if the route and prompt are still appropriate. |
| `cancel_failed` | Inspect `cancel_attempts`; do not broaden termination automatically. |
| `not_running` | Use `job-diagnose --format human` to inspect final state. |

## Long-running job failures

Separate these cases before retrying:

- Codex/Responses stream disconnect: UI or logs show `stream disconnected before completion` or reconnect attempts. Do not keep retrying built-in `image_gen` for complex prompts; switch to CLI `--background-job`.
- Shell outer timeout: output says `command timed out after ... milliseconds`. Increase the outer `timeout_ms` above CLI `--timeout` plus retries, or prefer `--background-job`.
- CLI internal API timeout: the Henry JSON envelope has `error.code: timeout`. Report the selected route, `auth_source`, and `candidate_attempts`; retry only if the next route/provider candidate is already authorized.
- External provider/API error: provider returns invalid auth, quota, policy, bad parameter, 5xx, or no image result. Report the exact blocker and do not modify proxy, `.env`, credentials, or AiMaMi config.
- Adaptive auth mismatch: inspect `auth_plan` in `probe`, `--dry-run`, `job-status --diagnose`, or `job-diagnose`. OpenAI-compatible should normally show Bearer; Azure/AOAI should show `api-key` and often `api-version`; local relay should show no-auth first. Logs list header/query names only and must not contain secret values.
- Private auth/schema required: OAuth refresh, browser cookie/session reuse, mTLS, HMAC/signature auth, or provider-private image payload/response schemas are out of scope for the generic adaptive layer. Report the blocker and use or build a provider-specific adapter.
- Detached child no-result: `job-status` returns `child_no_result` or `child_invalid_json`. Treat this as a job-runner/provider transport failure; report `stderr_tail` and best-effort `candidate_attempts`, then retry with `--background-job` only if the route/provider is still appropriate.
- Job cancellation: `job-cancel` only targets recorded `child_pid`, `runner_pid`, and `pid`. On Windows it does not use process-tree termination; on failure it returns `cancel_failed` with `cancel_attempts` for manual review.

## Job diagnosis and cancellation

For long or failed background jobs:

1. Run `job-status --job <job_id-or-path> --diagnose` to keep the JSON envelope and attach `outputs[0].diagnosis`.
2. Run `job-diagnose --job <job_id-or-path> --format human` for a readable Blocker / Evidence / Next action report.
3. If the job appears stuck, run `job-cancel --job <job_id-or-path> --dry-run` first. Only run `job-cancel` without `--dry-run` when the listed recorded PIDs are expected.

Diagnosis fields are `summary`, `category`, `next_action`, `evidence`, `files`, `attempts`, and `stderr_tail`. All output is redacted through the normal secret redaction path.

## Quality Failures

Classify these separately from API failures:

- wrong subject or missing required subject;
- ignored input-image role;
- copied feedback annotations, red boxes, UI chrome, or correction text;
- unreadable, wrong, or hallucinated text;
- dimension inconsistency;
- multi-view inconsistency;
- exposed internals when the design should be sealed;
- missing structural parts;
- implausible manufacturing geometry;
- reference-image drift;
- poor composition, blur, artifacts, or low quality.

For quality failures:

- ordinary visual defect: retry once with a targeted correction;
- role or annotation failure: restate input roles and remove annotation content;
- dimension or engineering failure: prefer deterministic SVG/PDF/OpenSCAD/spec;
- after two structural or dimension-critical failures, stop raster iteration and recommend deterministic output.

## Reporting

Keep failure reports short and concrete:

```text
Blocker: unsupported_router.
Evidence: route returned no image_generation result.
Next usable path: checked CLI fallback route; prompt package only if real image bytes cannot be produced or Henry asked for prompt-only output.
```

For quality failures:

```text
Quality failure: multi-view inconsistency.
Evidence: front view and top view show different cable-channel geometry.
Next usable path: deterministic SVG/PDF diagram.
```
